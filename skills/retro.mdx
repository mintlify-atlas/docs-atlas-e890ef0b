## What it does

`/retro` is your **engineering manager mode**. At the end of the week, you want to know what actually happened - not vibes, data.

This skill analyzes commit history, work patterns, and shipping velocity, then writes a candid retrospective. It's team-aware: identifies who's running the command, gives you the deepest treatment on your own work, then breaks down every contributor with specific praise and growth opportunities.

<Note>
Computes metrics like commits, LOC, test ratio, PR sizes, fix ratio, detects coding sessions from timestamps, finds hotspot files, tracks shipping streaks, and identifies the biggest ship of the week.
</Note>

## When to use it

Use `/retro` when you want to:
- Review the week's engineering work with data
- See team velocity, test coverage, and code quality trends
- Get specific feedback on each team member's contributions
- Track your personal coding patterns and peak hours
- Compare this week vs last week
- See your shipping streak

## Real example

From the README:

```bash

        ### Bob
        3 commits — fixed the N+1 query on dashboard. Small but high-impact.
        Opportunity: only 1 active day this week — check if blocked on anything.

        [Top 3 team wins, 3 things to improve, 3 habits for next week]
```

## Arguments

```bash
/retro              # last 7 days (default)
/retro 24h          # last 24 hours
/retro 14d          # last 14 days
/retro 30d          # last 30 days
/retro compare      # compare current window vs prior same-length window
/retro compare 14d  # compare with explicit window
```

## Team member analysis

For each contributor, the skill computes:

1. **Commits and LOC** - total commits, insertions, deletions, net LOC
2. **Areas of focus** - which directories/files they touched most (top 3)
3. **Commit type mix** - their personal feat/fix/refactor/test breakdown
4. **Session patterns** - when they code (peak hours), session count
5. **Test discipline** - their personal test LOC ratio
6. **Biggest ship** - their single highest-impact commit or PR

### Praise examples

<Note>
Praise is anchored in actual commits. Not "great work" - say exactly what was good.
</Note>

Examples:
- "Shipped the entire auth middleware rewrite in 3 focused sessions with 45% test coverage"
- "Every PR under 200 LOC - disciplined decomposition"
- "Fixed the N+1 query that was causing 2s load times on the dashboard"

### Growth opportunity examples

<Note>
Frame as leveling-up suggestions, not criticism. Anchor in actual data.
</Note>

Examples:
- "Test ratio was 12% this week - adding test coverage to the payment module before it gets more complex would pay off"
- "5 fix commits on the same file suggest the original PR could have used a review pass"
- "All commits land between 1-4am - sustainable pace matters for code quality long-term"

## Streak tracking

Counts consecutive days with at least 1 commit to `origin/main`, going back from today. Tracks both team streak and personal streak:

```bash
# Team streak: all unique commit dates (Pacific time)
TZ=America/Los_Angeles git log origin/main --format="%ad" --date=format:"%Y-%m-%d" | sort -u

# Personal streak: only the current user's commits
TZ=America/Los_Angeles git log origin/main --author="<user_name>" --format="%ad" --date=format:"%Y-%m-%d" | sort -u
```

Display both:
- "Team shipping streak: 47 consecutive days"
- "Your shipping streak: 32 consecutive days"

## Greptile signal tracking

If `~/.gstack/greptile-history.md` exists and has entries within the time window:

- Read entries and filter by date
- Count by type: `fix`, `fp`, `already-fixed`
- Compute signal ratio: `(fix + already-fixed) / (fix + already-fixed + fp)`
- Display: "Greptile: 83% signal (5 valid catches, 1 false positive)"

If no history exists, skip the Greptile metric row.

## Compare mode

When you run `/retro compare` or `/retro compare 14d`:

<Steps>
  <Step title="Compute current window metrics">
    Default 7d using `--since="7 days ago"`
  </Step>
  
  <Step title="Compute prior window metrics">
    Same-length window immediately before, using both `--since` and `--until` to avoid overlap.
    
    Example for 7d window:
    ```bash
    --since="14 days ago" --until="7 days ago"
    ```
  </Step>
  
  <Step title="Show side-by-side comparison">
    ```
                        Prior       Now         Delta
    Commits:            32     →    47          ↑47%
    Test ratio:         22%    →    41%         ↑19pp
    Sessions:           10     →    14          ↑4
    LOC/hour:           200    →    350         ↑75%
    Fix ratio:          54%    →    30%         ↓24pp (improving)
    Deep sessions:      3      →    5           ↑2
    ```
  </Step>
  
  <Step title="Write narrative">
    Highlight biggest improvements and regressions.
  </Step>
  
  <Step title="Save only current window">
    Save only the current-window snapshot to `.context/retros/` - do NOT persist prior-window metrics.
  </Step>
</Steps>

## Tips

<Accordion title="How often should I run /retro?">
Weekly is the default. Run it every Friday to review the week. If you're shipping fast, try `24h` or `3d` windows.
</Accordion>

<Accordion title="What if it's a solo repo?">
The skill detects this and skips the team breakdown. You get the full personal deep-dive instead.
</Accordion>

<Accordion title="Can I see trends over multiple weeks?">
Yes. Run `/retro compare` to see this week vs last week. Or use `14d` / `30d` windows to see weekly trends within the period.
</Accordion>

<Accordion title="What if the window has zero commits?">
The skill tells you and suggests a different window.
</Accordion>

## Tone

From the skill instructions:

- Encouraging but candid, no coddling
- Specific and concrete - always anchor in actual commits/code
- Skip generic praise ("great job!") - say exactly what was good and why
- Frame improvements as leveling up, not criticism
- **Praise should feel like something you'd actually say in a 1:1** - specific, earned, genuine
- **Growth suggestions should feel like investment advice** - "this is worth your time because..." not "you failed at..."
- Never compare teammates against each other negatively
